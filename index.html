    How the full analysis was calculated

    This page explains the full end-to-end logic used in the call analysis dashboard: what inputs were used, how every metric was created, which parts came from audio, which parts came from transcripts, what formulas were used, what rules were applied, and what the limits are.

    The analysis has two separate lanes: audio analysis from the recording signal itself, and conversation analysis from the transcript text. They are intentionally kept separate.

    What this methodology covers

    • Input files and folder structure
    • How calls were matched and renumbered batch-wise
    • How sound quality metrics were computed
    • How plain-language audio findings were assigned
    • How transcript summaries and drop stages were decided
    • How charts and batch rollups were built
    • What is measured directly vs inferred heuristically
    Input sources
    ZIP + transcripts + audio
    Analysis lanes
    2
    Audio lane and transcript lane
    Output layers
    3
    Call-level, batch-level, consolidated

    End-to-end pipeline

    Step 1: Input capture and pairing

    • The uploaded ZIP was extracted into batch folders.
    • Each subfolder inside a batch was treated as one call record.
    • Inside each call folder, the script looked for:
      • a transcript file such as transcription.csv or transcript.txt
      • an audio file such as .mp3
    • Calls were then renumbered from Recording 1 to Recording n within each batch, in ascending folder order.
    This means the displayed recording number is a presentation number for easier reading. It is not always the original folder name.

    Step 2: Two-lane separation

    LaneInputUsed forNot used for
    Audio laneMP3 waveformsilence, clarity, noise, background-speech sign, audio qualitymeaning of conversation, summary, intent
    Transcript laneCSV/TXT textsummary, user turn count, drop stage, transcript previewnoise judgment, sound quality, background interference scoring

    Audio analysis methodology

    The audio lane reads the recording signal, converts it to mono, breaks it into short windows, and computes summary values from the sound envelope and frequency texture. The UI then converts those technical values into plain-language scores and findings.

    Silence %

    Measures how much of the call is quiet relative to detected active speech or sound.

    silence_percent =
  100 × (1 - non_silent_duration / total_duration)
    • High value means lots of dead air, pauses, abandonment, or one-sided audio.
    • Used directly in call cards and in the batch silence chart.

    Longest silent stretch

    Measures the single longest continuous quiet segment in the file.

    longest_silence =
  max(duration of every detected silent segment)
    • Used to flag calls with one large inactivity block.

    Speech clarity score

    Represents how clearly speech stands above the quieter floor of the recording.

    raw_clarity =
  percentile_75(active_level) - percentile_10(quiet_floor)

clarity_score =
  clamp(0, 100, ((raw_clarity - 5) / 20) × 100)
    • Higher score means speech is easier to hear.
    • Lower score means speech and background are closer together.
    • The UI shows only the 0–100 score, not the underlying dB gap.

    Noise score

    Represents how noise-like the sound texture is across short frequency windows.

    spectral_flatness =
  geometric_mean(spectrum) / arithmetic_mean(spectrum)

noise_score =
  clamp(0, 100, spectral_flatness × 180)
    • Higher score means the sound is less speech-like and more noise-like.
    • This can reflect hiss, fan noise, room noise, traffic texture, or diffuse background sound.

    Volume-change score

    Measures how unstable the loudness is during the call.

    rms_cv =
  std(rms_windows) / mean(rms_windows)

volume_change_score =
  clamp(0, 100, (rms_cv / 1.5) × 100)
    • Higher score means stronger changes in loudness over time.
    • Can indicate changing distance from mic, inconsistent speech level, bursts, or mixed sources.

    Background-speech sign

    This is a heuristic indicator, not a direct detector. It estimates the likelihood that the call contains competing speech-like interference. 

    bg_speech_indicator =
  0.45 × noise_score
+ 0.35 × volume_change_score
+ 0.20 × (100 - clarity_score)

then clamp to 0–100
    • Higher score means stronger signs of competing or interfering sound.
    • This does not prove there was another speaker. It is a risk indicator only.

    How audio quality bucket was assigned

    The overall audio quality label is rule-based, using a weighted point system.

    score = 0

if silence >= 65: score += 2
else if silence >= 45: score += 1

if clarity <= 30: score += 2
else if clarity <= 55: score += 1

if clipping risk is high: score += 1
if noise score >= 60: score += 1

if score >= 4  -> Needs attention
if score >= 2  -> Okay
else           -> Clean

    This is not a machine-learning classifier. It is an explainable rule-based quality label.

    Transcript analysis methodology

    The transcript lane does not judge audio quality. It only interprets conversation structure.

    User turn count

    Counts the number of transcript rows whose sender is marked as user.

    user_turns =
  count(rows where sender == "user")

    Drop stage

    Assigned from user turn count only.

    if user_turns == 0:
  drop_stage = "Dropped at opening"
else if user_turns <= 2:
  drop_stage = "Dropped after short reply"
else:
  drop_stage = "Stayed for conversation"

    Conversation summary

    A short summary is generated using simple rule checks.

    if no user text:
  "call opened but no clear user response"

else if user denies appointment context:
  "appointment context did not match"

else if assistant says it couldn't catch user:
  "agent had trouble catching speech"

else if user_turns >= 3:
  "real back-and-forth but no reschedule"

else:
  "brief reply, conversation ended early"

    Transcript preview

    The UI preview shows the first part of the transcript only.

    preview =
  first 12 speaker-message lines

    Batch-level rollups

    Batch charts and tables are built by grouping calls by batch and averaging or counting the call-level metrics.

    Example rollup formulas

    batch_avg_silence =
  mean(call.silence_percent for calls in batch)

batch_avg_clarity =
  mean(call.clarity_score for calls in batch)

batch_avg_noise =
  mean(call.noise_score for calls in batch)

batch_opening_drops =
  count(calls where drop_stage == "Dropped at opening")

    Operational metrics

    The following values were not derived from recordings. They were taken from the user-provided batch summary:

    • calls attempted
    • calls picked
    • successful reschedules
    • average session duration
    • total session duration
    • opt-outs

    Programmatic logic overview

    1. Extract ZIP
2. Walk through batch folders
3. For each call folder:
   - load transcript if available
   - load audio if available
4. Renumber recordings inside each batch from 1..n
5. Run audio analysis:
   - convert to mono
   - split into short windows
   - calculate RMS envelope
   - detect silent and non-silent regions
   - calculate clarity, noise, instability, and background-speech sign
6. Run transcript analysis:
   - count user turns
   - assign drop stage
   - build rule-based summary
7. Build call-level records
8. Group calls by batch for rollups
9. Render charts, filters, and call explorer in HTML

    What is directly measured

    • audio duration
    • amount of silence
    • quiet-floor vs active-speech separation
    • noise-like texture level
    • volume instability
    • user turn count from transcript

    What is inferred heuristically

    • overall audio quality bucket
    • background-speech sign
    • plain-language findings like “possible competing speech”
    • conversation summary label

    Important limits

    • This is not speaker diarization. It does not identify who spoke when.
    • This is not source separation. It cannot prove whether the background sound came from TV, another person, traffic, or room echo.
    • Transcript-based logic depends on transcript quality. If the transcript misses speech, the summary may understate what happened.
    • Audio scores are designed to be understandable and explainable, not clinically exact.
    • The “background-speech sign” is a risk-style indicator created from multiple audio cues, not a direct detector.

    Bottom line

    The analysis system is deliberately explainable. Every label shown in the dashboard comes either from:

    • Direct metric a directly computed value from the waveform or transcript structure, or
    • Heuristic label a rule-based interpretation built from those values.

    That makes the dashboard easier to audit, revise, and trust.
